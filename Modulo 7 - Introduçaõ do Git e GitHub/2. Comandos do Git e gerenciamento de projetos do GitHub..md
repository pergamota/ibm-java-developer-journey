# Boas Práticas de Desenvolvimento com Git

Uma boa prática de desenvolvimento é:

- Criar uma **branch secundária**, que será uma cópia do código original.
    
- Fazer alterações nessa branch, e não diretamente na principal.
    

---

## Área de Preparação (Staging Area)

Após realizar alterações:

1. Selecionar os arquivos alterados.
    
2. Movê-los para uma **área de preparação (staging area)**.
    

Essa área:

- Armazena todas as alterações
    
- Permite revisar antes de salvar definitivamente
    
- Serve como etapa intermediária antes do commit
    

---

## Commit

Depois de adicionar os arquivos à área de preparação:

- Você confirma as alterações.
    
- Uma **mensagem de commit** fornece contexto.
    
- Essa mensagem ajuda você e sua equipe a entender o que foi alterado.
    

---

## Envio para o Repositório Remoto

Depois do commit:

- As alterações são enviadas para o repositório remoto.
    
- Antes de mesclar, ocorre um **code review**.
    
- Você cria uma **Pull Request** para que as mudanças sejam aprovadas.
    

A Pull Request:

- Permite revisão
    
- Permite comentários
    
- Verifica se está pronto para mesclar
    

---

# Fluxo Comum de Desenvolvimento

O fluxo comum é:

1. Sempre usar ramificações para alterações.
    
2. Solicitar testes.
    
3. Após aprovação, lançar no repositório remoto.
    

Mesmo que seja um projeto começando do zero:

1. Primeiro ocorre o desenvolvimento.
    
2. Depois uma análise.
    
3. Depois o commit.

# Comandos do Git

## mkdir

Significa **make directory**.

Cria um diretório.

---

## cd

Permite navegar entre diretórios através de caminhos.

---

## git init

Configura os arquivos e a estrutura de dados necessários para que o Git comece a gerenciar o controle de versão do projeto.

---

## git add

Move as alterações do diretório de trabalho para a área de preparação (staging area).

---

## git commit

Confirma as alterações adicionadas, junto com uma mensagem explicativa.

---

## git log

Revisa o histórico de commits.

---

## git checkout

Permite alternar entre ramificações existentes.

---

## git merge

Permite juntar e mesclar a ramificação de recursos com a ramificação principal.

---

## git fetch

Transfere quaisquer alterações do repositório remoto para o seu repositório local.

Não mescla automaticamente com sua branch atual.

---

## git pull

Transfere alterações do repositório remoto para o repositório local e mescla automaticamente com a branch atual.

---

## git request-pull

Cria um resumo das alterações para que o repositório upstream possa analisar e extrair as mudanças.

---

# Trabalhando com Ramificações

Para trabalhar com ramificações a partir do GitHub:

1. No repositório do GitHub, clique em **<> Code**.
    
2. Copie o link HTTPS.
    
3. No terminal, digite:
    

`git clone + HTTPS`

4. Entre no diretório do repositório usando `cd`.
    
5. Crie uma nova branch:
    

`git branch <nome-da-branch>`

Depois, você pode alternar para ela com:

`git checkout <nome-da-branch>`

---

# Clonagem (Clone)

Após clonar o repositório:

- Você terá uma cópia completa no seu computador.
    
- Depois disso, pode usar `git add` para mover arquivos alterados para a área de preparação.
    

## Área de Preparação

É onde os commits podem ser:

- Revisados
    
- Alterados
    
- Organizados antes da confirmação final
    

---

# Repositórios Remotos (Upstream)

Repositórios remotos são:

- Repositórios armazenados em outro local (ex: GitHub).
    
- Usados para compartilhar código.
    

É por meio deles que você:

1. Modifica o código localmente.
    
2. Usa `git push` para enviar alterações.
    
3. Apresenta o código para colegas.
    

---

# Bifurcação (Forking)

Fork é usado para:

- Criar uma cópia de um repositório do GitHub na sua própria conta.
    
- Usar como base para um novo projeto.
    
- Fazer alterações de forma independente.
    

Você usa o botão **Fork** no GitHub.

---

## Processo do Fork

1. Você cria um fork (que se torna sua origem).
    
2. Desenvolvedores criam clones locais.
    
3. Fazem alterações.
    
4. Usam `git push` para enviar alterações à branch principal do fork.
    
5. Criam Pull Requests para o repositório original.
    
6. As alterações são mescladas se não houver conflitos.
    

---

# Diferença entre Clone e Fork

- **Clone** → copia o repositório para seu computador.
    
- **Fork** → copia o repositório para sua conta no GitHub.

# Papéis no Projeto

## Integrador

O integrador:

- Recebe alterações feitas por desenvolvedores.
    
- Revisa.
    
- Integra as mudanças.
    
- Responde às solicitações de Pull.
    
- Publica o resultado final.
    

### Comandos usados pelo integrador:

#### git pull

Ajuda a fundir alterações recebidas.

#### git revert

Desfaz commits malsucedidos.

#### git push

Publica o projeto atualizado.

---

## Administrador

O administrador:

- Estrutura como o projeto será organizado.
    
- Define como usuários interagem com o repositório.
    
- Gerencia comunidades, relacionamentos, categorias e atributos.
    
- Configura e mantém acesso ao repositório.
    
- Define configurações de e-mail.
    
- Pode automatizar fluxos de trabalho.

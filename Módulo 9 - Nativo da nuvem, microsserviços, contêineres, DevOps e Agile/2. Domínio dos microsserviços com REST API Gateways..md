# Microserviços (Microservices)

## O que são?

É um estilo de arquitetura onde uma aplicação grande é dividida em **serviços pequenos, independentes e especializados**, que se comunicam entre si (geralmente via HTTP/REST).

✔ Cada serviço:

- É independente
- Tem sua própria lógica
- Pode ter seu próprio banco de dados
- Pode usar tecnologias diferentes

✔ Comunicação:

- APIs REST
- Streaming de eventos
- Mensageria

---

## Características

- Independência entre serviços
- Atualizações isoladas
- Equipes podem trabalhar separadamente
- Uso de múltiplas tecnologias

---

## Escalabilidade Horizontal

- Adiciona mais instâncias do serviço
- Cada microserviço pode escalar separadamente

✔ Exemplo:

- Serviço de login → 2 instâncias
- Serviço de pagamentos → 10 instâncias

---

## Desvantagens

- Mais difícil de depurar
- Maior complexidade
- Pode ter desafios de segurança

---

# Comparação de Arquiteturas

## 🔴 Monolítico

- Tudo em um único sistema
- Camadas interdependentes

✔ Vantagem:

- Simples

❌ Desvantagem:

- Difícil escalar
- Difícil manter com crescimento

---

## 🟡 SOA (Service-Oriented Architecture)

- Baseado em **provedor e consumidor de serviços**

### Componentes:

- Interface
- Contrato
- Implementação

✔ Vantagens:

- Reutilização
- Organização

❌ Desvantagens:

- Complexidade
- Alto custo

---

## 🟢 Microserviços

- Serviços pequenos e independentes
- Comunicação via API

✔ Vantagens:

- Escalabilidade
- Flexibilidade
- Deploy independente

---

# Padrões de Microserviços

## SPA (Single Page Application)

- Uma única página
- Não recarrega
- Melhor experiência no front-end

Pode sobrecarregar o back-end

---

## BFF (Backend for Frontend)

- Backend específico para cada tipo de cliente
- Ex: mobile vs web

Resolve problemas de múltiplos canais

---

## Strangler Pattern

- Migração gradual de monolito → microserviços
- Substitui partes antigas aos poucos

---

## Descoberta de Serviço

- Serviços se encontram dinamicamente
- Importante por causa da escalabilidade

Exemplo:

- Netflix Eureka

---

# REST (Representational State Transfer)

## O que é?

Estilo arquitetônico para comunicação entre sistemas.

---

## Características principais

- Usa HTTP
- Comunicação **cliente-servidor**
- **Sem estado (stateless)**
- Interface uniforme

---

## CRUD com HTTP

- **GET** → buscar
- **POST** → criar
- **PUT** → atualizar
- **DELETE** → remover

---

## Stateless

Cada requisição:

- Contém todas as informações necessárias
- Não depende de memória do servidor

---

## Interface Uniforme

- Cada recurso tem um **URI único**
- Ex: `/produtos/1`

---

# API Gateway

## O que é?

Camada intermediária entre cliente e microserviços.

---

## Funções

- Agregar serviços
- Rotear requisições
- Monitorar uso
- Proteger APIs
- Esconder complexidade do backend

---

## Benefícios

- Reduz número de requisições
- Padroniza comunicação
- Isola cliente da arquitetura interna

---

## Desvantagens

- Pode virar ponto único de falha
- Aumenta latência
- Mais complexidade

---

# Teste de APIs

## cURL

Ferramenta de linha de comando para requisições HTTP.

### Exemplo:

curl -X GET http://127.0.0.1:5000/products -H "accept: application/json"

### 📥 Resposta:

{  
  "id": 143,  
  "name": "Notebook"  
}

---

## Postman

- Interface gráfica
- Facilita testes de API
- Permite salvar requisições

---

# OpenAPI (Swagger)

## O que é?

Uma **especificação padrão para documentar APIs REST**.

Funciona como um **contrato da API**

---

## O que define?

- **Endpoints**
    - `/usuarios`
    - `/produtos/{id}`
- **Métodos HTTP**
    - GET, POST, PUT, DELETE
- **Parâmetros**
    - Query, Path, Header, Body
- **Respostas**
    - 200, 404, 500
- **Schemas**
    - Estrutura dos dados JSON
- **Autenticação**
    - JWT, OAuth2, API Key
- **Documentação**
    - Descrição e exemplos

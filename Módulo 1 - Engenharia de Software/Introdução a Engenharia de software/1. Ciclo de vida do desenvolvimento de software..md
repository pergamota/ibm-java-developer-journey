
# Engenharia de Software — Fundamentos

## Engenheiro de Software × Desenvolvedor de Software

### Engenheiro de Software

O engenheiro de software atua com uma **visão ampla do sistema**, envolvendo:

- Planejamento
    
- Arquitetura
    
- Design
    
- Tomada de decisões técnicas
    

Ele não se limita apenas à escrita de código, mas pensa no **software como um todo**, considerando escalabilidade, manutenção, custos e impacto para o usuário.


### Desenvolvedor de Software

O desenvolvedor de software é mais focado na **implementação prática**, atuando diretamente na escrita de código, geralmente:

- Backend
    
- Frontend
    
- Correção de bugs
    
- Implementação de funcionalidades
    

Ambos os papéis são complementares e essenciais dentro de um projeto.


## SDLC — Software Development Life Cycle

O **SDLC** é um modelo padronizado que orienta as equipes no desenvolvimento de software, garantindo:

- Qualidade
    
- Organização
    
- Cumprimento de prazos
    
- Controle de custos

### Fases do SDLC

1. **Planejamento**  
    Coleta e análise de requisitos.
    
2. **Arquitetura / Design**  
    Definição da arquitetura e criação dos documentos de design.
    
3. **Desenvolvimento**  
    Implementação do código.
    
4. **Testes**  
    Identificação e correção de defeitos.
    
5. **Implementação (Deploy)**  
    Liberação do software para produção.
    
6. **Manutenção**  
    Correções, melhorias e coleta de feedback.


## Documentos de Requisitos

### SRS — Software Requirements Specification

Documento que descreve:

- Requisitos funcionais
    
- Requisitos não funcionais
    
- Requisitos externos (interação usuário × sistema)


### URS — User Requirements Specification

Documento focado no usuário, contendo:

- Histórias de usuário
    
- Necessidades
    
- Expectativas de uso


### SysRS — System Requirements Specification

Define:

- Requisitos do sistema
    
- Critérios de aceitação
    
- Conformidade com feedbacks de stakeholders e usuários finais    


## Métodos de Engenharia de Software

### Cascata (Waterfall)

- Modelo **sequencial**
    
- Cada fase começa apenas após a anterior ser concluída
    

**Vantagens:**

- Fácil de entender
    
- Bem documentado
    

**Desvantagens:**

- Pouco flexível
    
- Difícil adaptação a mudanças


### Modelo em V

- Evolução do modelo cascata
    
- Lado esquerdo: **verificação**
    
- Lado direito: **validação**
    

Indicado para sistemas com requisitos bem definidos.


### Ágil

- Baseado em ciclos curtos (iterações)
    
- Forte comunicação com o cliente
    
- Feedback constante
    

**Vantagens:**

- Alta adaptabilidade
    
- Entrega contínua de valor
    

**Desvantagens:**

- Orçamento e prazo final menos previsíveis


## Versionamento de Software

Os softwares utilizam números de versão para indicar mudanças:

- **1º número** — Mudanças grandes (major)
    
- **2º número** — Melhorias menores (minor)
    
- **3º número** — Correções de bugs (patch)
    
- **4º número** — Build ou ajustes pouco significativos
    

## Tipos de Testes

### Testes Funcionais

- Baseados na **caixa preta**
    
- Avaliam entradas e saídas
    
- Verificam requisitos funcionais


### Testes Não Funcionais

Avaliam aspectos como:

- Desempenho
    
- Segurança
    
- Disponibilidade


### Teste de Regressão

Garante que alterações ou atualizações:

- Não quebrem funcionalidades existentes
    
- Não introduzam novos bugs


## Níveis de Teste

1. **Teste de Unidade**  
    Verifica componentes individuais do código.
    
2. **Teste de Integração**  
    Avalia a interação entre diferentes módulos.
    
3. **Teste de Sistema**  
    Verifica o sistema completo em relação aos requisitos.
    
4. **Teste de Aceitação**  
    Confirma se o sistema atende às expectativas do usuário.    


## Documentação de Software

### Tipos de Documentação

- **Documentação do Produto**  
    Funcionalidades e uso do software.
    
- **Documentação de Processo**  
    Como executar tarefas no sistema.
    
- **Documentação de Requisitos**  
    Direcionada à equipe de desenvolvimento.
    
- **Documentação de Design (Projeto)**  
    Explica como o software será construído.
    
- **Documentação Técnica**  
    Comentários no código e detalhes técnicos.
    
- **Documentação de Usuário**  
    Guia de uso do sistema.
    
- **SOP (Standard Operating Procedure)**  
    Passo a passo para tarefas complexas.


## Papéis em Projetos de Software

### Gerente de Projeto / Scrum Master

Responsável por:

- Planejamento
    
- Comunicação
    
- Acompanhamento do projeto


### Stakeholders

Pessoas interessadas no projeto que:

- Definem requisitos
    
- Fornecem feedbacks


### Arquiteto de Software

Responsável por:

- Arquitetura do sistema
    
- Decisões técnicas
    
- Estrutura interna do software


### UX Designer

Focado na:

- Experiência do usuário
    
- Usabilidade
    
- Interface intuitiva


### QA Engineer / Testers

Garantem:

- Qualidade do produto
    
- Conformidade com requisitos


### Site Reliability Engineer (SRE)

Atuam em:

- Automação
    
- Confiabilidade
    
- Resolução de incidentes


### Product Manager / Product Owner

Definem:

- Visão do produto
    
- Prioridades
    
- Necessidades do usuário


### Technical Writer / Information Developer

Responsáveis por:

- Documentação técnica
    
- Conteúdo para o usuário final


## Observação Final

A engenharia de software envolve **processos, pessoas e tecnologia**, garantindo que o software seja confiável, escalável e alinhado às necessidades do negócio.
